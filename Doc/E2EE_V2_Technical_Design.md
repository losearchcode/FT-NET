# FT-NET E2EE v2 技术草案

## 1. 目标

本草案用于定义 FT-NET 下一阶段端到端加密（E2EE v2）的技术落地方案。目标是在保留当前“密码即房间”“统一端口”“服务端中转存储”架构优势的前提下，将现有的“部分加密能力”升级为：

- 更现代的密码学实现
- 更明确的密钥分工
- 更低的服务端明文暴露面
- 可兼容现有数据与后续协议演进的版本化设计

本草案优先解决以下问题：

- 当前文本消息与文件加密方案不统一，且依赖 `CryptoJS` 的旧式封装方式
- 文件名、发送者等元数据仍存在明文暴露
- 缺少协议版本字段，不利于后续迭代升级
- 文件“加密上传 / 明文直传”虽已支持，但尚未形成统一的安全模型

### 当前落地状态（2026-04）

- 已完成：消息加密 v2
  - 新消息支持 `Web Crypto API + PBKDF2 + HKDF + AES-GCM`
  - 已接入 `v1 / v2` 双读兼容
  - 已加入房间能力协商，避免不支持环境误收 `v2` 消息
- 已完成：文件内容加密 v2
  - 新加密文件支持分片 `AES-GCM`
  - 上传端已支持按能力选择 `v2 / v1 / 明文` 三条路径
  - 下载与预览端已支持按文件元数据自动双读 `v2 / v1`
- 已完成：文件元数据加密
  - `fileName` 等敏感字段已保存在服务端加密元数据中，物理存储仅保留随机化 id 和 `encryptedMetadata`
- 未完成：完整房间级 E2EE 收口
  - 仍需补齐统一密钥策略增强、恢复能力与更清晰的 UI 安全提示

## 2. 设计原则

### 2.1 零信任服务端

服务端只负责：

- 房间管理
- WebSocket 中转
- 文件分块接收与顺序落盘
- 加密元数据的存储与分发

服务端不应掌握：

- 聊天消息明文
- 加密文件内容明文
- 加密文件文件名等敏感元数据明文

### 2.2 版本化协议

所有新写入的加密数据都必须带有 `version` 字段。

建议格式：

- `v1`：当前已存在的 `CryptoJS` 方案
- `v2`：新方案（Web Crypto API + AES-GCM + 独立密钥分工）

客户端读取历史数据时，应根据 `version` 自动选择解密路径。

### 2.3 明确的模式边界

房间通信与文件传输应当明确区分：

- **消息默认加密**
- **文件可选加密**

文件模式建议统一标识为：

- `securityMode: encrypted`
- `securityMode: plain`

客户端必须根据该字段决定展示、下载和预览流程。

## 3. 密钥体系设计

### 3.1 输入

房间口令仍然作为唯一共享秘密输入：

- `roomPassword`

### 3.2 主密钥派生

使用 Web Crypto API 的 PBKDF2 从房间口令派生主密钥：

- 算法：`PBKDF2`
- 哈希：`SHA-256`
- 输出：`256 bit`
- 迭代次数：建议 `200000+`
- 盐值：
  - 短期兼容方案：固定项目盐 + 协议版本盐
  - 后续增强方案：房间级随机盐，通过服务端分发公开值

派生结果记为：

- `masterKey`

### 3.3 子密钥分工

由 `masterKey` 再派生出多个用途隔离的子密钥：

- `messageKey`
  - 用于聊天消息加解密
- `fileKey`
  - 用于文件内容分块加解密
- `metadataKey`
  - 用于文件元数据加解密

建议做法：

- 通过 HKDF 或“主密钥 + purpose label”再次派生
- 避免所有内容共用同一把对称密钥

---

## 4. 消息加密设计

### 4.1 加密算法

消息加密统一迁移到：

- `AES-GCM`

原因：

- 同时提供机密性与完整性保护
- 原生支持认证标签
- 适合 Web Crypto API

### 4.2 消息密文结构

建议消息结构：

```json
{
  "version": "v2",
  "type": "TEXT",
  "securityMode": "encrypted",
  "payload": {
    "iv": "<base64>",
    "ciphertext": "<base64>"
  },
  "senderName": "<optional or encrypted in future>",
  "timestamp": 1710000000000
}
```

说明：

- `iv` 每条消息独立随机生成
- `ciphertext` 为 AES-GCM 输出
- `senderName` 第一阶段可暂保留明文，后续可纳入元数据加密

### 4.3 解密策略

客户端收到消息后：

1. 检查 `version`
2. 若为 `v2`，使用 `messageKey` 解密
3. 若为 `v1`，走兼容解密分支

## 5. 文件加密设计

### 5.1 文件模式

文件统一分为两种：

- `encrypted`
- `plain`

文件元数据中必须显式标注：

```json
{
  "securityMode": "encrypted"
}
```

### 5.2 加密文件内容

建议继续保留当前的“分块上传 / 顺序落盘”架构，但将分块密文格式升级到 `v2`。

每个分片独立加密：

- 算法：`AES-GCM`
- 密钥：`fileKey`
- 每片独立 `iv`
- 每片附带认证标签

优点：

- 更适合失败重试
- 更适合断点续传
- 更适合流式下载与边解密边写盘

当前状态：

- 已落地
- 当前实现已在新文件加密上传中启用 `AES-GCM`
- 历史 `v1` 文件继续保留兼容读取能力
- 明文文件仍按 `plain` 模式直传与直下

### 5.3 分片密文结构

建议上传分片结构：

```json
{
  "version": "v2",
  "securityMode": "encrypted",
  "chunkIndex": 12,
  "iv": "<base64>",
  "ciphertext": "<binary or base64>",
  "plainSize": 4194304
}
```

实际传输时可采用“二进制头 + 二进制密文”形式，减少 JSON 开销。

### 5.4 明文文件内容

若文件为 `plain`：

- 不进行内容加密
- 继续使用当前分块上传协议
- 服务端直接顺序写入最终文件

## 6. 文件元数据加密设计

### 6.1 需要保护的字段

建议第一阶段优先保护：

- `fileName`

后续可继续扩展：

- `senderName`
- `customLabels`

短期内可保留明文字段：

- `uploadedAt`
- `fileSize`
- `securityMode`

因为这些字段对列表展示和下载流程有现实价值。

### 6.2 元数据结构

建议将敏感元数据单独打包后加密：

```json
{
  "version": "v2",
  "metadata": {
    "fileName": "合同.pdf"
  }
}
```

再使用 `metadataKey` 做 AES-GCM 加密，服务端最终存储：

```json
{
  "id": "server-physical-id",
  "securityMode": "encrypted",
  "fileSize": 123456,
  "uploadedAt": 1710000000000,
  "senderId": "abc123",
  "encryptedMetadata": {
    "version": "v2",
    "iv": "<base64>",
    "ciphertext": "<base64>"
  }
}
```

这样服务端即使拿到文件列表，也无法直接看到原始文件名。

## 7. 浏览器与运行环境策略

### 7.1 Web Crypto API

E2EE v2 的核心实现建议以浏览器原生 Web Crypto API 为主。

原因：

- 更现代
- 更安全
- 性能优于纯 JS 库
- 与 AES-GCM / PBKDF2 / HKDF 更契合

### 7.2 流式保存

下载侧继续维持当前双路径：

- 支持 `showSaveFilePicker`：
  - 边下载
  - 边解密
  - 边写入本地文件
- 不支持：
  - 回退到 Blob 内存模式

### 7.3 安全上下文

流式保存与部分现代能力受浏览器安全上下文限制：

- 推荐环境：`HTTPS`
- 本机环境：`localhost`
- 局域网 `HTTP`：通常会被浏览器视为不安全上下文

该限制属于浏览器安全策略，不属于项目逻辑错误。

## 8. 兼容与迁移方案

### 8.1 兼容原则

系统需要同时兼容：

- 已有 `v1` 消息
- 已有 `v1` 加密文件
- 新写入的 `v2` 消息与文件

### 8.2 迁移策略

建议采用“只增不改”的渐进迁移：

- 老数据保留原格式
- 新数据默认写入 `v2`
- 读取端根据 `version` 自适应解密

不建议一开始就对服务端历史数据做离线重写。

---

## 9. 实施顺序建议

### 阶段 A：消息加密 v2

优先实现：

- Web Crypto API
- `PBKDF2 + AES-GCM`
- 文本消息 `version = v2`
- `v1 / v2` 双读

原因：

- 改动面小
- 最容易验证
- 最快形成稳定样板

### 阶段 B：文件元数据加密

实现：

- `fileName` 加密
- 文件列表解密展示
- 服务端存储 `encryptedMetadata`

当前状态：

- 已完成
- 客户端在接收文件时已能够自动利用 metadataKey 解密展示原始文件名

### 阶段 C：文件分块加密 v2

实现：

- 文件内容分块 AES-GCM
- 每片独立 `iv`
- 下载端分片解密
- 与当前顺序写盘协议结合

当前状态：

- 已完成核心落地
- 已新增独立的 `encryptWorkerV2 / decryptWorkerV2`
- 已支持新文件写入 `v2`
- 已支持下载端根据文件版本自动选择 `v1 / v2` 解密路径
- 后续仍可继续增强分片校验、断点续传与元数据保护

### 阶段 D：体验与恢复能力

实现：

- 断点续传
- 分片校验
- 更强的错误恢复
- 更明确的安全状态提示

## 10. 风险与注意事项

### 10.1 房间密码强度

若继续沿用“房间密码即密钥根输入”模型，则用户输入弱密码时：

- 仍存在被穷举风险

因此 UI 层建议后续增加：

- 最低密码长度提示
- 弱密码警告

### 10.2 元数据完全隐藏的代价

完全隐藏文件大小、时间戳、发送者等字段会提高隐私性，但也会显著增加：

- 列表渲染复杂度
- 下载流程复杂度
- 调试成本

因此建议分阶段推进，而非一次性全隐藏。

### 10.3 明文模式的长期存在

明文直传模式是现实需求，但必须在产品文案中持续明确：

- 它不是 E2EE
- 它是性能优先模式

避免用户将“进入加密房间”误解为“所有文件必然是密文”。

后续实现时建议优先改造以下位置：

- `src/utils/cryptoUtils.ts`
  - 抽出 `v1 / v2` 双版本能力
- `src/hooks/usePeer.ts`
  - 接管消息加密、文件上传模式与版本写入
- `src/components/FileTransfer.tsx`
  - 接管文件列表解密展示、下载模式分流
- `src/workers/encryptWorker.ts`
  - 升级分片加密实现
- `src/workers/decryptWorker.ts`
  - 升级分片解密实现
- `server/index.js`
  - 适配 `encryptedMetadata` 与文件模式字段

## 12. 结论

FT-NET 当前已经具备向“完整 E2EE”演进的基础：

- 房间密码模型已经存在
- 文本与文件的前端加解密链路已经存在
- 文件上传已具备分块协议基础
- 下载端已具备流式保存能力分流

下一步的重点不是“再增加一点加密”，而是建立一套：

- 现代的加密算法实现
- 清晰的密钥分工
- 受保护的元数据结构
- 可版本化演进的协议体系

本草案建议以“消息 v2 → 元数据加密 → 文件 v2 → 恢复能力增强”的顺序推进，以降低风险并提高落地效率。
