# FT-NET 安全升级规划：端到端加密 (E2EE) 需求文档

## 1. 当前安全痛点 (V1.0)
当前 FT-NET 采用了以 `31208` 端口为核心的统一网络托管架构。该版架构虽然绝对高效地穿透了跨平台防火墙，但在**底层防解密层面处于纯明文 (“裸奔”) 状态**：
- **传输层网络嗅探**：所有的 HTTP / WebSocket 数据流在局域网路由器抓包工具面前均等同于明码暴露。
- **存储层权限黑洞**：Node.js 服务端在物理硬盘 (`/uploads` 目录) 以及内存 Map 数据库 (`rooms`) 中保存的文字及文件均为未加盖戳的“全裸态”。即使只是被攻破了一个外网低危漏洞，拿到了硬盘权限的非房管黑客也可直接全数拖走敏感数据。

## 2. 第一阶段方案：底层传输网络加密 (TLS 加壳)
- **防御目标**：防中间人 (MITM)、同路由隔壁节点网络抓包窃听。
- **加锁逻辑**：
  强制在云服务器环境接入 Nginx 反向代理层。必须通过免费第三方 (如 Let's Encrypt 等) 签发企业级 SSL 证书，强行将入口由 `http://` 转换锁定为 443 加密的 `https://` 并支持 `wss://`。

## 3. 第二阶段究极防卫方案：零信任端到端加密 (全流 E2EE)
- **防御目标**：“连服务器系统底层和创造者也绝对无法偷窥还原的终极黑箱”。
- **演进路线蓝图**：
  我们目前极简的【暗号密码进房】是一个绝佳的突破口！在下一次安全叠代中，将毫不犹豫直接把这串用于开门的“纯密码”直接转换为高级对称加密框架 (`AES-256`) 的顶级本地运算公匙：
  1. **消息加密**：在用户的 React/浏览器层，监听按下“发送”的回调，利用 `crypto-js` 第三方黑盒调用【房间密码】充当密匙在本地强行“绞碎”文字转为加密散列乱码再做 POST。3001 内存里停留的所有对话将等同废卡。
  2. **二进制资源流加密**：引入最强大的 HTML5 Web Crypto API。任何扔进左侧【通讯柜】里的图片、图纸或是视频文件流 ArrayBuffer，全部由本地 CPU 高速压缩运算进行 AES-GCM 封装成废文件后再传至云端 `/uploads`。
  3. **只认令牌提取**：唯有另一个知晓同样“接头暗号”密码的人加入此房抓取了“这一段废物乱码和图包”后，他手里的前端 JS CPU 大脑才会再次利用这把共同的 AES 暗盒钥匙逆向进行数据流的实时显形脱壳和视觉还原。彻底从根源抹除后端硬盘监守自盗的基础条件！
